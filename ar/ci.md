# CI Pipeline

يعمل CI عند كل عملية push إلى `main` وكل pull request. يستخدم تحديد نطاق ذكيًا لتخطي الوظائف المكلفة عندما يتم تغيير الوثائق أو الشيفرة الأصلية فقط.

## نظرة عامة على الوظائف

| الوظيفة           | الغرض                                                                   | متى يتم تشغيله                                 |
| ----------------- | ----------------------------------------------------------------------- | ---------------------------------------------- |
| `docs-scope`      | اكتشاف التغييرات التي تقتصر على المستندات فقط                           | دائمًا                                         |
| `changed-scope`   | اكتشاف المجالات التي تم تغييرها (node/macos/android) | طلبات السحب التي لا تتعلق بالمستندات           |
| `check`           | التحقق من أنواع TypeScript، وفحص lint، والتنسيق                         | تغييرات غير متعلقة بالمستندات                  |
| `check-docs`      | فحص Markdown + التحقق من الروابط المعطلة                                | تم تغيير المستندات                             |
| `code-analysis`   | التحقق من حد عدد الأسطر (1000 سطر)                   | طلبات السحب فقط                                |
| `secrets`         | اكتشاف الأسرار المسربة                                                  | دائمًا                                         |
| `build-artifacts` | بناء dist مرة واحدة ومشاركته مع المهام الأخرى                           | تغييرات غير متعلقة بالمستندات، تغييرات node    |
| `release-check`   | التحقق من محتويات npm pack                                              | بعد البناء                                     |
| `checks`          | اختبارات Node/Bun + التحقق من البروتوكول                                | تغييرات غير متعلقة بالمستندات، تغييرات node    |
| `checks-windows`  | اختبارات خاصة بـ Windows                                                | تغييرات غير متعلقة بالمستندات، تغييرات node    |
| `macos`           | فحص وبناء واختبار Swift + اختبارات TS                                   | طلبات السحب التي تتضمن تغييرات macos           |
| `android`         | بناء Gradle + اختبارات                                                  | تغييرات غير متعلقة بالمستندات، تغييرات android |

## ترتيب الإخفاق السريع

تم ترتيب المهام بحيث تفشل عمليات الفحص منخفضة التكلفة قبل تشغيل العمليات المكلفة:

1. `docs-scope` + `code-analysis` + `check` (بالتوازي، ~1-2 دقيقة)
2. `build-artifacts` (محجوب اعتمادًا على ما سبق)
3. `checks`، `checks-windows`، `macos`، `android` (محجوبة حتى اكتمال build)

## Runners

| Runner                          | Jobs                                          |
| ------------------------------- | --------------------------------------------- |
| `blacksmith-4vcpu-ubuntu-2404`  | معظم مهام Linux                               |
| `blacksmith-4vcpu-windows-2025` | `checks-windows`                              |
| `macos-latest`                  | `macos`، `ios`                                |
| `ubuntu-latest`                 | اكتشاف النطاق (خفيف الوزن) |

## المكافئات المحلية

```bash
pnpm check          # الأنواع + lint + التنسيق
pnpm test           # اختبارات vitest
pnpm check:docs     # تنسيق التوثيق + lint + الروابط المعطلة
pnpm release:check  # التحقق من npm pack
```
