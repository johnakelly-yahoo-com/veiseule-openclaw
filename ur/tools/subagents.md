# ذیلی ایجنٹس

سب ایجنٹس وہ بیک گراؤنڈ ایجنٹ رنز ہیں جو کسی موجودہ ایجنٹ رن سے شروع کیے جاتے ہیں۔ یہ اپنی علیحدہ سیشن (`agent:<agentId>:subagent:<uuid>`) میں چلتے ہیں اور مکمل ہونے پر اپنی نتیجہ کی **اطلاع** درخواست کنندہ کے چیٹ چینل پر دیتے ہیں۔

## Slash کمانڈ

**موجودہ سیشن** کے لیے سب ایجنٹ رنز کا جائزہ لینے یا کنٹرول کرنے کے لیے `/subagents` استعمال کریں:

- `/subagents list`
- `/subagents kill <id|#|all>`
- `/subagents log <id|#> [limit] [tools]`
- `/subagents info <id|#>`
- `/subagents send <id|#> <message>`

The simplest way to use sub-agents is to ask your agent naturally:

بنیادی مقاصد:

- مرکزی رن کو بلاک کیے بغیر "research / طویل کام / سست ٹول" جیسے کاموں کو متوازی بنانا۔
- سب ایجنٹس کو بطور ڈیفالٹ علیحدہ رکھنا (سیشن علیحدگی + اختیاری sandboxing)۔
- ٹول سرفیس کو غلط استعمال سے محفوظ رکھنا: سب ایجنٹس کو بطور ڈیفالٹ سیشن ٹولز نہیں ملتے۔
- orchestrator پیٹرنز کے لیے قابلِ کنفیگریشن nesting depth کی حمایت۔

لاگت نوٹ: ہر سب ایجنٹ کا اپنا **الگ** context اور ٹوکن استعمال ہوتا ہے۔ بھاری یا بار بار ہونے والے
کاموں کے لیے سب ایجنٹس پر کم قیمت ماڈل سیٹ کریں اور اپنے مرکزی ایجنٹ کو اعلیٰ معیار کے ماڈل پر رکھیں۔
آپ اسے `agents.defaults.subagents.model` یا فی ایجنٹ overrides کے ذریعے کنفیگر کر سکتے ہیں۔

## ٹول

`sessions_spawn` استعمال کریں:

- ایک سب ایجنٹ رن شروع کرتا ہے (`deliver: false`, global lane: `subagent`)
- پھر announce مرحلہ چلاتا ہے اور announce جواب درخواست کنندہ کے چیٹ چینل پر پوسٹ کرتا ہے
- ڈیفالٹ ماڈل: کالر سے وراثت میں لیتا ہے جب تک آپ `agents.defaults.subagents.model` (یا فی ایجنٹ `agents.list[].subagents.model`) سیٹ نہ کریں؛ واضح طور پر دیا گیا `sessions_spawn.model` پھر بھی حاوی رہے گا۔
- ڈیفالٹ thinking: کالر سے وراثت میں لیتا ہے جب تک آپ `agents.defaults.subagents.thinking` (یا فی ایجنٹ `agents.list[].subagents.thinking`) سیٹ نہ کریں؛ واضح طور پر دیا گیا `sessions_spawn.thinking` پھر بھی حاوی رہے گا۔

ٹول پیرامیٹرز:

- `task` (لازمی)
- `label?` (اختیاری)
- `agentId?` (اختیاری؛ اگر اجازت ہو تو کسی دوسرے ایجنٹ id کے تحت شروع کریں)
- `model?` (اختیاری؛ سب ایجنٹ ماڈل کو اووررائیڈ کرتا ہے؛ غلط ویلیوز کو نظر انداز کر دیا جاتا ہے اور سب ایجنٹ ڈیفالٹ ماڈل پر چلے گا جبکہ ٹول کے نتیجے میں وارننگ ہوگی)
- `thinking?` (اختیاری؛ سب ایجنٹ رن کے لیے thinking لیول اووررائیڈ کرتا ہے)
- `runTimeoutSeconds?` (ڈیفالٹ `0`؛ سیٹ ہونے پر سب ایجنٹ رن N سیکنڈ کے بعد منسوخ کر دیا جاتا ہے)
- `cleanup?` (`delete|keep`، ڈیفالٹ `keep`)

Allowlist:

- `agents.list[].subagents.allowAgents`: ان agent ids کی فہرست جنہیں `agentId` کے ذریعے ہدف بنایا جا سکتا ہے (`["*"]` کسی بھی agent کی اجازت دینے کے لیے)۔ Default: صرف درخواست کرنے والا agent۔

Discovery:

- یہ دیکھنے کے لیے `agents_list` استعمال کریں کہ اس وقت کون سے agent ids کو `sessions_spawn` کے لیے اجازت حاصل ہے۔

Auto-archive:

- Sub-agent سیشنز خودکار طور پر `agents.defaults.subagents.archiveAfterMinutes` (پہلے سے طے شدہ: 60) کے بعد آرکائیو ہو جاتے ہیں۔
- Archive میں `sessions.delete` استعمال ہوتا ہے اور ٹرانسکرپٹ کا نام تبدیل کر کے `*.deleted.<timestamp>` (اسی فولڈر میں)۔
- `cleanup: "delete"` اعلان کے فوراً بعد آرکائیو کر دیتا ہے (ٹرانسکرپٹ کو نام تبدیل کر کے محفوظ رکھتا ہے)۔
- Auto-archive بہترین کوشش کی بنیاد پر کام کرتا ہے؛ اگر gateway دوبارہ شروع ہو جائے تو زیرِ التوا ٹائمر ضائع ہو جاتے ہیں۔
- `runTimeoutSeconds` **خودکار آرکائیو نہیں کرتا**؛ یہ صرف run کو روکتا ہے۔ سیشن Auto-archive تک برقرار رہتا ہے۔
- Auto-archive کا اطلاق depth-1 اور depth-2 سیشنز دونوں پر یکساں ہوتا ہے۔

## Nested Sub-Agents

پہلے سے طے شدہ طور پر، sub-agents اپنے ذاتی sub-agents اسپون نہیں کر سکتے (`maxSpawnDepth: 1`)۔ آپ `maxSpawnDepth: 2` سیٹ کر کے ایک سطح کی nesting فعال کر سکتے ہیں، جو **orchestrator pattern** کی اجازت دیتا ہے: main → orchestrator sub-agent → worker sub-sub-agents۔

### فعال کرنے کا طریقہ

```json5
{
  agents: {
    list: [
      {
        id: "researcher",
        subagents: {
          model: "anthropic/claude-sonnet-4",
        },
      },
      {
        id: "assistant",
        subagents: {
          model: "minimax/MiniMax-M2.1",
        },
      },
    ],
  },
}
```

### ہم وقتیّت

| Depth | Session key کی ساخت                          | کردار                                                                 | کیا spawn کر سکتا ہے؟           |
| ----- | -------------------------------------------- | --------------------------------------------------------------------- | ------------------------------- |
| 0     | `agent:&lt;id&gt;:main`                            | Main agent                                                            | ہمیشہ                           |
| 1     | `agent:&lt;id&gt;:subagent:<uuid>`                 | Sub-agent (جب depth 2 کی اجازت ہو تو orchestrator) | صرف اگر `maxSpawnDepth >= 2` ہو |
| 2     | `agent:&lt;id&gt;:subagent:<uuid>:subagent:<uuid>` | Sub-sub-agent (leaf worker)                        | کبھی نہیں                       |

### Announce چین

سب-ایجنٹس ایک مخصوص کیو لین (`subagent`) استعمال کرتے ہیں جو مین ایجنٹ کیو سے الگ ہوتی ہے، اس لیے سب-ایجنٹ رنز آنے والے جوابات کو بلاک نہیں کرتے۔

1. Depth-2 worker مکمل کرتا ہے → اپنے parent (depth-1 orchestrator) کو announce کرتا ہے
2. Depth-1 orchestrator announce وصول کرتا ہے، نتائج کو یکجا کرتا ہے، مکمل کرتا ہے → main کو announce کرتا ہے
3. Main agent announce وصول کرتا ہے اور صارف کو فراہم کرتا ہے

سب-ایجنٹ سیشنز ایک قابلِ ترتیب مدت کے بعد خودکار طور پر آرکائیو ہو جاتے ہیں:

### Tool پالیسی بر اساس depth

- **Depth 1 (orchestrator، جب `maxSpawnDepth >= 2`)**: اسے `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history` ملتے ہیں تاکہ وہ اپنے children کو منظم کر سکے۔ دیگر session/system ٹولز بدستور ممنوع رہتے ہیں۔
- **گہرائی 1 (لیف، جب `maxSpawnDepth == 1` ہو)**: کوئی سیشن ٹولز نہیں (موجودہ ڈیفالٹ رویہ).
- **گہرائی 2 (لیف ورکر)**: کوئی سیشن ٹولز نہیں — `sessions_spawn` کو گہرائی 2 پر ہمیشہ مسترد کیا جاتا ہے۔ مزید بچوں کو اسپان نہیں کیا جا سکتا۔

### `sessions_spawn` ٹول

ہر ایجنٹ سیشن (کسی بھی گہرائی پر) ایک وقت میں زیادہ سے زیادہ `maxChildrenPerAgent` (ڈیفالٹ: 5) فعال بچوں کا حامل ہو سکتا ہے۔ یہ ایک ہی آرکیسٹریٹر سے بے قابو پھیلاؤ کو روکتا ہے۔

### Parameters

گہرائی-1 آرکیسٹریٹر کو روکنے سے اس کے تمام گہرائی-2 بچے خود بخود رک جاتے ہیں:

- مرکزی چیٹ میں `/stop` تمام گہرائی-1 ایجنٹس کو روکتا ہے اور یہ عمل ان کے گہرائی-2 بچوں تک پھیل جاتا ہے۔
- `/subagents kill <id>` کسی مخصوص ذیلی ایجنٹ کو روکتا ہے اور اس کے بچوں تک بھی یہ عمل پھیلتا ہے۔
- `/subagents kill all` درخواست گزار کے تمام ذیلی ایجنٹس کو روکتا ہے اور یہ عمل آگے تک پھیلتا ہے۔

## تصدیق

ذیلی ایجنٹ کی تصدیق **ایجنٹ آئی ڈی** کی بنیاد پر حل ہوتی ہے، نہ کہ سیشن کی قسم پر:

- ذیلی ایجنٹ سیشن کی کلید `agent:<agentId>:subagent:<uuid>` ہے۔
- auth اسٹور اس ایجنٹ کے `agentDir` سے لوڈ کیا جاتا ہے۔
- مرکزی ایجنٹ کے auth پروفائلز کو **fallback** کے طور پر ضم کیا جاتا ہے؛ تنازع کی صورت میں ایجنٹ کے پروفائلز مرکزی پروفائلز پر فوقیت رکھتے ہیں۔

نوٹ: یہ ضم عمل اضافی نوعیت کا ہے، اس لیے مرکزی پروفائلز ہمیشہ fallback کے طور پر دستیاب رہتے ہیں۔ ہر ایجنٹ کے لیے مکمل طور پر الگ تھلگ auth ابھی تک سپورٹ نہیں کیا جاتا۔

## اعلان

ذیلی ایجنٹس ایک اعلان مرحلے کے ذریعے رپورٹ واپس کرتے ہیں:

- اعلان مرحلہ ذیلی ایجنٹ سیشن کے اندر چلتا ہے (درخواست گزار کے سیشن میں نہیں)۔
- اگر ذیلی ایجنٹ عین `ANNOUNCE_SKIP` جواب دے تو کچھ بھی پوسٹ نہیں کیا جاتا۔
- بصورتِ دیگر اعلان کا جواب درخواست گزار کے چیٹ چینل میں فالو اَپ `agent` کال (`deliver=true`) کے ذریعے پوسٹ کیا جاتا ہے۔
- اعلان کے جوابات دستیاب ہونے پر تھریڈ/موضوع کی روٹنگ برقرار رکھتے ہیں (Slack تھریڈز، Telegram ٹاپکس، Matrix تھریڈز)۔
- اعلان پیغامات کو ایک مستحکم ٹیمپلیٹ میں معیاری بنایا جاتا ہے:
  - `Status:` رن کے نتیجے سے اخذ کیا جاتا ہے (`success`، `error`، `timeout`، یا `unknown`)۔
  - `Result:` اعلان مرحلے کے خلاصہ مواد پر مشتمل ہوتا ہے (یا اگر موجود نہ ہو تو `(not available)`)۔
  - `Notes:` خرابی کی تفصیلات اور دیگر مفید سیاق و سباق۔
- `Status` ماڈل کے آؤٹ پٹ سے اخذ نہیں کیا جاتا؛ یہ رن ٹائم کے نتائج کے سگنلز سے آتا ہے۔

اعلان payloads کے آخر میں ایک اسٹیٹس لائن شامل ہوتی ہے (چاہے لپٹی ہوئی ہو):

- رن ٹائم (مثلاً، `runtime 5m12s`)
- ٹوکن استعمال (ان پٹ/آؤٹ پٹ/کل)
- جب ماڈل پرائسنگ کنفیگر ہو تو تخمینی لاگت (`models.providers.*.models[].cost`)
- `sessionKey`، `sessionId`، اور ٹرانسکرپٹ کا راستہ (تاکہ مرکزی ایجنٹ `sessions_history` کے ذریعے ہسٹری حاصل کر سکے یا ڈسک پر فائل کا معائنہ کر سکے)

## ٹول پالیسی (ذیلی ایجنٹ ٹولز)

ڈیفالٹ کے طور پر، ذیلی ایجنٹس کو **سیشن ٹولز کے علاوہ تمام ٹولز** اور سسٹم ٹولز دیے جاتے ہیں:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

## ہم وقتیّت

ذیلی ایجنٹس ایک مخصوص in-process قطار لین استعمال کرتے ہیں:

- لین کا نام: `subagent`
- بیک وقت عمل درآمد: `agents.defaults.subagents.maxConcurrent` (ڈیفالٹ `8`)

## روکنا

- ریکوئسٹر چیٹ میں `/stop` بھیجنے سے ریکوئسٹر سیشن منسوخ ہو جاتا ہے اور اس سے شروع کیے گئے تمام فعال سب-ایجنٹ رنز بھی رک جاتے ہیں، اور یہ عمل نیسٹڈ بچوں تک بھی پھیل جاتا ہے۔
- `/subagents kill <id>` کسی مخصوص سب-ایجنٹ کو روکتا ہے اور اس کے بچوں تک بھی یہ عمل پھیل جاتا ہے۔

## حدود

- سب-ایجنٹ اعلان **best-effort** بنیاد پر ہوتا ہے۔ اگر gateway دوبارہ شروع ہو جائے تو زیرِ التواء "announce back" کام ضائع ہو جاتا ہے۔
- سب-ایجنٹس اب بھی وہی gateway پراسیس وسائل استعمال کرتے ہیں؛ `maxConcurrent` کو حفاظتی والو کے طور پر استعمال کریں۔
- `sessions_spawn` ہمیشہ non-blocking ہوتا ہے: یہ فوراً `{ status: "accepted", runId, childSessionKey }` واپس کرتا ہے۔
- سب-ایجنٹ کانٹیکسٹ میں صرف `AGENTS.md` + `TOOLS.md` شامل کیے جاتے ہیں (`SOUL.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`، یا `BOOTSTRAP.md` شامل نہیں ہوتے)۔
- زیادہ سے زیادہ nesting گہرائی 5 ہے (`maxSpawnDepth` کی حد: 1–5)۔ زیادہ تر استعمال کے معاملات کے لیے گہرائی 2 تجویز کی جاتی ہے۔
- `maxChildrenPerAgent` ہر سیشن کے لیے فعال بچوں کی حد مقرر کرتا ہے (ڈیفالٹ: 5، حد: 1–20)۔
