# ذیلی ایجنٹس

ذیلی ایجنٹس ایسے بیک گراؤنڈ ایجنٹ رنز ہیں جو کسی موجودہ ایجنٹ رن سے شروع کیے جاتے ہیں۔ یہ اپنی علیحدہ سیشن (`agent:<agentId>:subagent:<uuid>`) میں چلتے ہیں اور مکمل ہونے پر اپنا نتیجہ درخواست کنندہ چیٹ چینل میں **announce** کرتے ہیں۔

## سلیش کمانڈ

موجودہ سیشن کے لیے ذیلی ایجنٹ رنز کو دیکھنے یا کنٹرول کرنے کے لیے `/subagents` استعمال کریں:

- `/subagents list`
- `/subagents kill <id|#|all>`
- `/subagents log <id|#> [limit] [tools]`
- `/subagents info <id|#>`
- `/subagents send <id|#> <message>`

`/subagents info` رن میٹاڈیٹا دکھاتا ہے (اسٹیٹس، ٹائم اسٹیمپس، سیشن آئی ڈی، ٹرانسکرپٹ پاتھ، صفائی)۔

اہم مقاصد:

- مرکزی رن کو بلاک کیے بغیر "ریسرچ / طویل کام / سست ٹول" کو متوازی بنانا۔
- ذیلی ایجنٹس کو بطور ڈیفالٹ الگ تھلگ رکھنا (سیشن علیحدگی + اختیاری سینڈ باکسنگ)۔
- ٹول سطح کو غلط استعمال سے محفوظ رکھنا: ذیلی ایجنٹس کو بطور ڈیفالٹ سیشن ٹولز نہیں ملتے۔
- آرکیسٹریٹر پیٹرنز کے لیے قابلِ ترتیب نیسٹنگ گہرائی کی حمایت۔

لاگت کا نوٹ: ہر ذیلی ایجنٹ کا اپنا **الگ** کانٹیکسٹ اور ٹوکن استعمال ہوتا ہے۔ بھاری یا بار بار ہونے والے کاموں کے لیے ذیلی ایجنٹس پر سستا ماڈل سیٹ کریں اور مرکزی ایجنٹ کو اعلیٰ معیار کے ماڈل پر رکھیں۔ آپ یہ `agents.defaults.subagents.model` یا فی-ایجنٹ اوور رائیڈز کے ذریعے ترتیب دے سکتے ہیں۔

## ٹول

`sessions_spawn` استعمال کریں:

- ایک ذیلی ایجنٹ رن شروع کرتا ہے (`deliver: false`, گلوبل لین: `subagent`)
- پھر announce مرحلہ چلاتا ہے اور announce جواب درخواست کنندہ چیٹ چینل میں پوسٹ کرتا ہے
- ڈیفالٹ ماڈل: کالر سے وراثت لیتا ہے جب تک آپ `agents.defaults.subagents.model` (یا `agents.list[].subagents.model`) سیٹ نہ کریں؛ واضح `sessions_spawn.model` پھر بھی غالب رہے گا۔
- ڈیفالٹ سوچ: کالر سے وراثت لیتی ہے جب تک آپ `agents.defaults.subagents.thinking` (یا `agents.list[].subagents.thinking`) سیٹ نہ کریں؛ واضح `sessions_spawn.thinking` پھر بھی غالب رہے گا۔

ٹول پیرامیٹرز:

- `task` (درکار)
- `label?` (اختیاری)
- `agentId?` (اختیاری؛ اجازت ہونے پر کسی دوسرے ایجنٹ آئی ڈی کے تحت اسپان کریں)
- `model?` (اختیاری؛ ذیلی ایجنٹ ماڈل اوور رائیڈ؛ غلط ویلیوز کو چھوڑ دیا جاتا ہے اور ذیلی ایجنٹ ڈیفالٹ ماڈل پر ایک وارننگ کے ساتھ چلتا ہے)
- `thinking?` (اختیاری؛ ذیلی ایجنٹ رن کے لیے سوچ کی سطح اوور رائیڈ)
- `runTimeoutSeconds?` (ڈیفالٹ `0`؛ سیٹ ہونے پر N سیکنڈ بعد رن منسوخ)
- `cleanup?` (`delete|keep`, ڈیفالٹ `keep`)

Allowlist:

- `agents.list[].subagents.allowAgents`: ایجنٹ آئی ڈیز کی فہرست جنہیں `agentId` کے ذریعے ہدف بنایا جا سکتا ہے (`["*"]` کسی بھی کی اجازت دینے کے لیے)۔ ڈیفالٹ: صرف درخواست کنندہ ایجنٹ۔

Discovery:

- دیکھنے کے لیے کہ اس وقت کون سی ایجنٹ آئی ڈیز `sessions_spawn` کے لیے مجاز ہیں، `agents_list` استعمال کریں۔

Auto-archive:

- ذیلی ایجنٹ سیشنز خودکار طور پر `agents.defaults.subagents.archiveAfterMinutes` (ڈیفالٹ: 60) کے بعد آرکائیو ہو جاتے ہیں۔
- آرکائیو `sessions.delete` استعمال کرتا ہے اور ٹرانسکرپٹ کا نام `*.deleted.<timestamp>` میں تبدیل کرتا ہے (اسی فولڈر میں)۔
- `cleanup: "delete"` announce کے فوراً بعد آرکائیو کر دیتا ہے (ٹرانسکرپٹ rename کے ذریعے محفوظ رہتی ہے)۔
- خودکار آرکائیو بہترین کوشش کی بنیاد پر ہے؛ اگر گیٹ وے دوبارہ شروع ہو جائے تو زیرِ التواء ٹائمرز ضائع ہو جاتے ہیں۔
- `runTimeoutSeconds` خودکار آرکائیو نہیں کرتا؛ یہ صرف رن روکتا ہے۔ سیشن auto-archive تک برقرار رہتا ہے۔
- auto-archive گہرائی-1 اور گہرائی-2 دونوں سیشنز پر یکساں لاگو ہوتا ہے۔

## نیسٹڈ ذیلی ایجنٹس

بطور ڈیفالٹ، ذیلی ایجنٹس اپنے مزید ذیلی ایجنٹس اسپان نہیں کر سکتے (`maxSpawnDepth: 1`)۔ آپ `maxSpawnDepth: 2` سیٹ کر کے ایک سطح کی نیسٹنگ فعال کر سکتے ہیں، جو **orchestrator pattern** کی اجازت دیتا ہے: main → orchestrator sub-agent → worker sub-sub-agents۔

### فعال کرنے کا طریقہ

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2, // ذیلی ایجنٹس کو بچوں کو اسپان کرنے کی اجازت (ڈیفالٹ: 1)
        maxChildrenPerAgent: 5, // ہر ایجنٹ سیشن کے لیے زیادہ سے زیادہ فعال بچے (ڈیفالٹ: 5)
        maxConcurrent: 8, // گلوبل ہم وقتی حد (ڈیفالٹ: 8)
      },
    },
  },
}
```

### گہرائی کی سطحیں

| Depth | Session key shape                            | Role                                          | Can spawn?                   |
| ----- | -------------------------------------------- | --------------------------------------------- | ---------------------------- |
| 0     | `agent:<id>:main`                            | مرکزی ایجنٹ                                  | ہمیشہ                       |
| 1     | `agent:<id>:subagent:<uuid>`                 | ذیلی ایجنٹ (جب depth 2 مجاز ہو تو orchestrator) | صرف اگر `maxSpawnDepth >= 2` |
| 2     | `agent:<id>:subagent:<uuid>:subagent:<uuid>` | ذیلی-ذیلی ایجنٹ (leaf worker)                | کبھی نہیں                    |

### Announce سلسلہ

نتائج زنجیر کے ذریعے واپس آتے ہیں:

1. Depth-2 ورکر مکمل ہوتا ہے → اپنے پیرنٹ (depth-1 orchestrator) کو announce کرتا ہے
2. Depth-1 orchestrator announce وصول کر کے نتائج کو یکجا کرتا ہے، مکمل ہوتا ہے → main کو announce کرتا ہے
3. Main ایجنٹ announce وصول کر کے صارف کو ڈیلیور کرتا ہے

ہر سطح صرف اپنے براہِ راست بچوں کے announces دیکھتی ہے۔

### گہرائی کے لحاظ سے ٹول پالیسی

- **Depth 1 (orchestrator، جب `maxSpawnDepth >= 2`)**: `sessions_spawn`, `subagents`, `sessions_list`, `sessions_history` حاصل کرتا ہے تاکہ اپنے بچوں کو مینیج کر سکے۔ دیگر سیشن/سسٹم ٹولز منع رہتے ہیں۔
- **Depth 1 (leaf، جب `maxSpawnDepth == 1`)**: کوئی سیشن ٹول نہیں (موجودہ ڈیفالٹ رویہ)۔
- **Depth 2 (leaf worker)**: کوئی سیشن ٹول نہیں — `sessions_spawn` ہمیشہ depth 2 پر منع ہے۔ مزید بچے اسپان نہیں کر سکتا۔

### فی-ایجنٹ اسپان حد

ہر ایجنٹ سیشن (کسی بھی گہرائی پر) بیک وقت زیادہ سے زیادہ `maxChildrenPerAgent` (ڈیفالٹ: 5) فعال بچے رکھ سکتا ہے۔ یہ ایک ہی orchestrator سے بے قابو فین آؤٹ کو روکتا ہے۔

### Cascade stop

کسی depth-1 orchestrator کو روکنا اس کے تمام depth-2 بچوں کو بھی خودکار طور پر روکتا ہے:

- مین چیٹ میں `/stop` تمام depth-1 ایجنٹس کو روکتا ہے اور ان کے depth-2 بچوں تک کاسکیڈ کرتا ہے۔
- `/subagents kill <id>` مخصوص ذیلی ایجنٹ کو روکتا ہے اور اس کے بچوں تک کاسکیڈ کرتا ہے۔
- `/subagents kill all` درخواست کنندہ کے تمام ذیلی ایجنٹس کو روکتا ہے اور کاسکیڈ کرتا ہے۔

## تصدیق

ذیلی ایجنٹ کی تصدیق **ایجنٹ آئی ڈی** کی بنیاد پر حل ہوتی ہے، نہ کہ سیشن کی قسم پر:

- ذیلی ایجنٹ سیشن کی `agent:<agentId>:subagent:<uuid>` ہوتی ہے۔
- auth اسٹور اسی ایجنٹ کے `agentDir` سے لوڈ ہوتا ہے۔
- مرکزی ایجنٹ کے auth پروفائلز کو **fallback** کے طور پر ضم کیا جاتا ہے؛ تنازع کی صورت میں ایجنٹ پروفائلز غالب رہتے ہیں۔

نوٹ: یہ ضم کرنا additive ہے، اس لیے مرکزی پروفائلز ہمیشہ fallback کے طور پر دستیاب رہتے ہیں۔ مکمل طور پر الگ تھلگ auth فی ایجنٹ فی الحال معاونت یافتہ نہیں۔

## Announce

ذیلی ایجنٹس announce مرحلے کے ذریعے رپورٹ کرتے ہیں:

- announce مرحلہ ذیلی ایجنٹ سیشن کے اندر چلتا ہے (درخواست کنندہ سیشن میں نہیں)۔
- اگر ذیلی ایجنٹ بالکل `ANNOUNCE_SKIP` جواب دے، تو کچھ بھی پوسٹ نہیں ہوتا۔
- ورنہ announce جواب ایک follow-up `agent` کال (`deliver=true`) کے ذریعے درخواست کنندہ چیٹ چینل میں پوسٹ کیا جاتا ہے۔
- announce جوابات دستیاب ہونے پر تھریڈ/موضوع روٹنگ برقرار رکھتے ہیں (Slack threads, Telegram topics, Matrix threads)۔
- announce پیغامات کو ایک مستحکم ٹیمپلیٹ میں نارملائز کیا جاتا ہے:
  - `Status:` رن کے نتیجے سے اخذ (`success`, `error`, `timeout`, یا `unknown`)۔
  - `Result:` announce مرحلے کا خلاصہ مواد (یا `(not available)` اگر موجود نہ ہو)۔
  - `Notes:` غلطی کی تفصیلات اور دیگر مفید سیاق۔
- `Status` ماڈل آؤٹ پٹ سے اخذ نہیں کیا جاتا؛ یہ رن ٹائم سگنلز سے آتا ہے۔

announce payloads کے آخر میں ایک stats لائن شامل ہوتی ہے (حتیٰ کہ wrapped ہونے پر بھی):

- Runtime (مثلاً `runtime 5m12s`)
- Token usage (input/output/total)
- تخمینی لاگت جب ماڈل پرائسنگ `models.providers.*.models[].cost` میں ترتیب دی گئی ہو
- `sessionKey`, `sessionId`, اور ٹرانسکرپٹ پاتھ (تاکہ main ایجنٹ `sessions_history` کے ذریعے ہسٹری حاصل کر سکے یا فائل براہِ راست دیکھ سکے)

## Tool Policy (ذیلی ایجنٹ ٹولز)

بطور ڈیفالٹ، ذیلی ایجنٹس کو **تمام ٹولز ملتے ہیں سوائے سیشن ٹولز** اور سسٹم ٹولز کے:

- `sessions_list`
- `sessions_history`
- `sessions_send`
- `sessions_spawn`

جب `maxSpawnDepth >= 2` ہو، تو depth-1 orchestrator ذیلی ایجنٹس اضافی طور پر `sessions_spawn`, `subagents`, `sessions_list`, اور `sessions_history` حاصل کرتے ہیں تاکہ اپنے بچوں کو مینیج کر سکیں۔

کنفیگریشن کے ذریعے اوور رائیڈ کریں:

```json5
{
  agents: {
    defaults: {
      subagents: {
        maxConcurrent: 1,
      },
    },
  },
  tools: {
    subagents: {
      tools: {
        // deny غالب رہتا ہے
        deny: ["gateway", "cron"],
        // اگر allow سیٹ ہو، تو یہ allow-only بن جاتا ہے (deny پھر بھی غالب رہے گا)
        // allow: ["read", "exec", "process"]
      },
    },
  },
}
```

## ہم وقتیّت

ذیلی ایجنٹس ایک مخصوص in-process queue lane استعمال کرتے ہیں:

- Lane name: `subagent`
- Concurrency: `agents.defaults.subagents.maxConcurrent` (ڈیفالٹ `8`)

## روکنا

- درخواست کنندہ چیٹ میں `/stop` بھیجنے سے درخواست کنندہ سیشن منسوخ ہو جاتا ہے اور اس سے اسپان ہونے والے تمام فعال ذیلی ایجنٹ رنز رک جاتے ہیں، نیسٹڈ بچوں تک کاسکیڈ کرتے ہوئے۔
- `/subagents kill <id>` مخصوص ذیلی ایجنٹ کو روکتا ہے اور اس کے بچوں تک کاسکیڈ کرتا ہے۔

## حدود

- ذیلی ایجنٹ announce **best-effort** ہے۔ اگر گیٹ وے دوبارہ شروع ہو جائے تو زیرِ التواء "announce back" کام ضائع ہو جاتا ہے۔
- ذیلی ایجنٹس اب بھی وہی گیٹ وے پروسیس وسائل شیئر کرتے ہیں؛ `maxConcurrent` کو حفاظتی حد کے طور پر استعمال کریں۔
- `sessions_spawn` ہمیشہ non-blocking ہے: یہ فوراً `{ status: "accepted", runId, childSessionKey }` واپس کرتا ہے۔
- ذیلی ایجنٹ کانٹیکسٹ صرف `AGENTS.md` + `TOOLS.md` inject کرتا ہے (کوئی `SOUL.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`, یا `BOOTSTRAP.md` نہیں)۔
- زیادہ سے زیادہ نیسٹنگ گہرائی 5 ہے (`maxSpawnDepth` حد: 1–5)۔ زیادہ تر استعمالات کے لیے depth 2 تجویز کردہ ہے۔
- `maxChildrenPerAgent` فی سیشن فعال بچوں کو محدود کرتا ہے (ڈیفالٹ: 5، حد: 1–20)۔
