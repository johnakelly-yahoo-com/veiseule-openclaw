# CI-конвейер

CI запускается при каждом push в `main` и при каждом pull request. Он использует интеллектуальное определение области изменений, чтобы пропускать ресурсоёмкие задачи, если изменились только docs или нативный код.

## Обзор задач

| Задача            | Назначение                                                              | Когда запускается                               |
| ----------------- | ----------------------------------------------------------------------- | ----------------------------------------------- |
| `docs-scope`      | Определение изменений только в документации                             | Всегда                                          |
| `changed-scope`   | Определение изменённых областей (node/macos/android) | PR без изменений в документации                 |
| `check`           | Типы TypeScript, lint, форматирование                                   | Изменения не в документации                     |
| `check-docs`      | Проверка Markdown (lint) + проверка битых ссылок     | Изменена документация                           |
| `code-analysis`   | Проверка порога LOC (1000 строк)                     | Только PR                                       |
| `secrets`         | Обнаружение утекших секретов                                            | Всегда                                          |
| `build-artifacts` | Собрать dist один раз и использовать в других задачах                   | Изменения не в документации, изменения в node   |
| `release-check`   | Проверка содержимого npm pack                                           | После сборки                                    |
| `checks`          | Тесты Node/Bun + проверка протокола                                     | Изменения не в документации, изменения в node   |
| `checks-windows`  | Тесты, специфичные для Windows                                          | Изменения не в документации, изменения в node   |
| `macos`           | Swift lint/сборка/тесты + тесты TS                                      | PR с изменениями в macos                        |
| `android`         | Сборка Gradle + тесты                                                   | Изменения вне документации, изменения в android |

## Порядок Fail-Fast

Задачи упорядочены так, чтобы недорогие проверки завершались ошибкой до запуска ресурсоёмких:

1. `docs-scope` + `code-analysis` + `check` (параллельно, ~1–2 мин)
2. `build-artifacts` (блокируется предыдущими)
3. `checks`, `checks-windows`, `macos`, `android` (блокируются сборкой)

## Раннеры

| Раннер                          | Задачи                                               |
| ------------------------------- | ---------------------------------------------------- |
| `blacksmith-4vcpu-ubuntu-2404`  | Большинство Linux-задач                              |
| `blacksmith-4vcpu-windows-2025` | `checks-windows`                                     |
| `macos-latest`                  | `macos`, `ios`                                       |
| `ubuntu-latest`                 | Определение области (лёгковесное) |

## Локальные эквиваленты

```bash
pnpm check          # типы + lint + форматирование
pnpm test           # тесты vitest
pnpm check:docs     # форматирование документации + lint + битые ссылки
pnpm release:check  # проверка npm pack
```
