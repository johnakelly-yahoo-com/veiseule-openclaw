# План рефакторинга Browser Evaluate CDP

## Контекст

`act:evaluate` выполняет предоставленный пользователем JavaScript на странице. В настоящее время он выполняется через Playwright
(`page.evaluate` или `locator.evaluate`). Playwright сериализует команды CDP для каждой страницы, поэтому
зависший или долго выполняющийся evaluate может заблокировать очередь команд страницы и сделать так, что все последующие действия
в этой вкладке будут выглядеть «зависшими».

PR #13498 добавляет практический механизм защиты (ограниченный по времени evaluate, распространение abort и попытка
восстановления по мере возможности). Этот документ описывает более масштабный рефакторинг, который делает `act:evaluate` изначально
изолированным от Playwright, чтобы зависший evaluate не мог блокировать обычные операции Playwright.

## Цели

- `act:evaluate` не может навсегда блокировать последующие действия браузера в той же вкладке.
- Тайм-ауты являются единым источником истины сквозным образом, чтобы вызывающая сторона могла полагаться на заданный бюджет.
- Abort и тайм-аут обрабатываются одинаково как для HTTP, так и для внутрипроцессной диспетчеризации.
- Таргетинг элемента для evaluate поддерживается без полного отказа от Playwright.
- Сохранить обратную совместимость для существующих вызовов и payload’ов.

## Не цели

- Заменить все действия браузера (click, type, wait и т.д.) на реализации через CDP.
- Удалить существующий механизм защиты, добавленный в PR #13498 (он остаётся полезным резервным вариантом).
- Вводить новые небезопасные возможности сверх существующего ограничения `browser.evaluateEnabled`.
- Добавить изоляцию процесса (worker process/thread) для evaluate. Если после этого рефакторинга мы всё ещё будем сталкиваться с трудновосстанавливаемыми
  зависшими состояниями, это станет задачей для следующего этапа.

## Текущая архитектура (почему возникают зависания)

В общих чертах:

- Вызывающие стороны отправляют `act:evaluate` в сервис управления браузером.
- Обработчик маршрута вызывает Playwright для выполнения JavaScript.
- Playwright сериализует команды страницы, поэтому evaluate, который никогда не завершается, блокирует очередь.
- Зависшая очередь означает, что последующие операции click/type/wait во вкладке могут выглядеть как зависшие.

## Предлагаемая архитектура

### 1. Передача дедлайна

Ввести единое понятие бюджета и выводить всё из него:

- Вызывающая сторона задаёт `timeoutMs` (или дедлайн в будущем).
- Внешний таймаут запроса, логика обработчика маршрута и бюджет выполнения внутри страницы
  используют один и тот же бюджет с небольшим запасом на накладные расходы сериализации, где это необходимо.
- Прерывание распространяется в виде `AbortSignal` повсюду, чтобы отмена была согласованной.

Направление реализации:

- Добавить небольшой хелпер (например, `createBudget({ timeoutMs, signal })`), который возвращает:
  - `signal`: связанный AbortSignal
  - `deadlineAtMs`: абсолютный дедлайн
  - `remainingMs()`: оставшийся бюджет для дочерних операций
- Использовать этот хелпер в:
  - `src/browser/client-fetch.ts` (HTTP и внутрипроцессная диспетчеризация)
  - `src/node-host/runner.ts` (путь через прокси)
  - реализациях действий браузера (Playwright и CDP)

### 2. Отдельный движок Evaluate (путь CDP)

Добавить реализацию evaluate на основе CDP, которая не использует общую для страницы очередь команд Playwright. Ключевое свойство заключается в том, что транспорт evaluate — это отдельное WebSocket-соединение
и отдельная CDP-сессия, подключённая к цели.

Направление реализации:

- Новый модуль, например `src/browser/cdp-evaluate.ts`, который:
  - Подключается к настроенному CDP-эндпоинту (сокет уровня браузера).
  - Использует `Target.attachToTarget({ targetId, flatten: true })` для получения `sessionId`.
  - Выполняет одно из:
    - `Runtime.evaluate` для выполнения evaluate на уровне страницы, или
    - `DOM.resolveNode` плюс `Runtime.callFunctionOn` для выполнения evaluate на уровне элемента.
  - При таймауте или прерывании:
    - Отправляет `Runtime.terminateExecution` для сессии в режиме best-effort.
    - Закрывает WebSocket и возвращает понятную ошибку.

Примечания:

- Это по-прежнему выполняет JavaScript на странице, поэтому принудительное завершение может иметь побочные эффекты. Преимущество
  в том, что это не блокирует очередь Playwright и может быть отменено на транспортном
  уровне путём завершения CDP-сессии.

### 3. Ref Story (Таргетирование элементов без полной переработки)

Самая сложная часть — таргетирование элементов. CDP требует DOM-хэндл или `backendDOMNodeId`, тогда как
сегодня большинство действий в браузере используют локаторы Playwright на основе ref из снапшотов.

Рекомендуемый подход: сохранить существующие ref, но добавить необязательный CDP-совместимый id.

#### 3.1 Расширение сохраняемой информации о ref

Расширить сохраняемые метаданные role ref, чтобы при необходимости включать CDP id:

- Сегодня: `{ role, name, nth }`
- Предлагается: `{ role, name, nth, backendDOMNodeId?: number }`

Это сохраняет работоспособность всех существующих действий на основе Playwright и позволяет CDP evaluate принимать
tо же значение `ref`, когда доступен `backendDOMNodeId`.

#### 3.2 Заполнение backendDOMNodeId во время создания снапшота

При формировании role-снапшота:

1. Сгенерировать существующую карту role ref, как и сейчас (role, name, nth).
2. Получить AX-дерево через CDP (`Accessibility.getFullAXTree`) и вычислить параллельную карту
   `(role, name, nth) -> backendDOMNodeId`, используя те же правила обработки дубликатов.
3. Объединить id обратно в сохранённую информацию о ref для текущей вкладки.

Если сопоставление для ref не удалось, оставить `backendDOMNodeId` неопределённым. Это делает функцию
реализованной в режиме best-effort и безопасной для поэтапного внедрения.

#### 3.3 Поведение evaluate с Ref

В `act:evaluate`:

- Если `ref` присутствует и содержит `backendDOMNodeId`, выполнить evaluate элемента через CDP.
- Если `ref` присутствует, но не содержит `backendDOMNodeId`, использовать путь через Playwright (с
  механизмом защиты).

Необязательный обходной путь:

- Расширить структуру запроса, чтобы принимать `backendDOMNodeId` напрямую для продвинутых пользователей (и
  для отладки), при этом сохранив `ref` как основной интерфейс.

### 4. Сохранить путь восстановления в крайнем случае

Даже при использовании CDP evaluate существуют другие способы «заблокировать» вкладку или соединение. Сохранить
существующие механизмы восстановления (terminate execution + отключение Playwright) как крайний вариант
для:

- устаревших вызовов
- сред, где подключение CDP заблокировано
- непредвиденных пограничных случаев Playwright

## План реализации (одна итерация)

### Результаты

- Движок evaluate на основе CDP, работающий вне очереди команд Playwright для конкретной страницы.
- Единый сквозной бюджет тайм-аута/прерывания, последовательно используемый вызывающими сторонами и обработчиками.
- Метаданные ref, которые при необходимости могут содержать `backendDOMNodeId` для evaluate элемента.
- `act:evaluate` по возможности использует движок CDP и при необходимости переключается на Playwright.
- Тесты, подтверждающие, что зависший evaluate не блокирует последующие действия.
- Логи/метрики, делающие сбои и переходы на резервный механизм видимыми.

### Контрольный список реализации

1. Добавить общий вспомогательный модуль "budget" для связывания `timeoutMs` и вышестоящего `AbortSignal` в:
   - единый `AbortSignal`
   - абсолютный дедлайн
   - вспомогательную функцию `remainingMs()` для нижележащих операций
2. Обновить все пути вызова для использования этого вспомогательного модуля, чтобы `timeoutMs` означал одно и то же везде:
   - `src/browser/client-fetch.ts` (HTTP и внутрипроцессная диспетчеризация)
   - `src/node-host/runner.ts` (путь через node-прокси)
   - CLI-обёртки, вызывающие `/act` (добавить `--timeout-ms` в `browser evaluate`)
3. Реализовать `src/browser/cdp-evaluate.ts`:
   - подключение к сокету CDP уровня браузера
   - `Target.attachToTarget` для получения `sessionId`
   - выполнение `Runtime.evaluate` для evaluate страницы
   - выполнение `DOM.resolveNode` + `Runtime.callFunctionOn` для evaluate элемента
   - при тайм-ауте/прерывании: попытаться выполнить `Runtime.terminateExecution`, затем закрыть сокет
4. Расширить сохраняемые метаданные role ref, добавив при необходимости `backendDOMNodeId`:
   - сохранить существующее поведение `{ role, name, nth }` для действий Playwright
   - добавить `backendDOMNodeId?: number` для таргетинга элемента через CDP
5. Заполнять `backendDOMNodeId` при создании snapshot (по возможности):
   - получать AX-дерево через CDP (`Accessibility.getFullAXTree`)
   - вычислять соответствие `(role, name, nth) -> backendDOMNodeId` и объединять с сохранённой картой ref
   - если сопоставление неоднозначно или отсутствует, оставить id неопределённым
6. Обновить маршрутизацию `act:evaluate`:
   - если `ref` отсутствует: всегда использовать evaluate через CDP
   - если `ref` сопоставляется с `backendDOMNodeId`: использовать evaluate элемента через CDP
   - в противном случае: переключаться на evaluate через Playwright (также с ограничением по времени и поддержкой прерывания)
7. Сохранить существующий путь восстановления «на крайний случай» как резервный, а не как основной.
8. Добавить тесты:
   - зависший evaluate завершается по тайм-ауту в пределах бюджета, и следующий click/type выполняется успешно
   - прерывание отменяет evaluate (разрыв соединения клиента или тайм-аут) и разблокирует последующие действия
   - сбои сопоставления корректно переключаются на Playwright
9. Добавить наблюдаемость:
   - метрики длительности evaluate и счётчики тайм-аутов
   - использование terminateExecution
   - коэффициент отката (CDP -> Playwright) и причины

### Критерии приемки

- Преднамеренно зависший `act:evaluate` возвращается в пределах бюджета вызывающей стороны и не блокирует вкладку для последующих действий.
- `timeoutMs` ведёт себя согласованно в CLI, инструменте агента, node proxy и во внутрипроцессных вызовах.
- Если `ref` можно сопоставить с `backendDOMNodeId`, evaluate элемента использует CDP; в противном случае путь отката остаётся ограниченным по времени и восстанавливаемым.

## План тестирования

- Модульные тесты:
  - Логика сопоставления `(role, name, nth)` между role-ссылками и узлами AX-дерева.
  - Поведение помощника управления бюджетом (запас по времени, расчёт оставшегося времени).
- Интеграционные тесты:
  - Тайм-аут CDP evaluate укладывается в бюджет и не блокирует следующее действие.
  - Abort отменяет evaluate и инициирует завершение в режиме best-effort.
- Контрактные тесты:
  - Убедиться, что `BrowserActRequest` и `BrowserActResponse` остаются совместимыми.

## Риски и меры по их снижению

- Сопоставление несовершенно:
  - Митигирование: сопоставление в режиме best-effort, откат к Playwright evaluate и добавление инструментов отладки.
- `Runtime.terminateExecution` имеет побочные эффекты:
  - Митигирование: использовать только при тайм-ауте/abort и документировать это поведение в ошибках.
- Дополнительные накладные расходы:
  - Митигирование: запрашивать AX-дерево только при необходимости снимков, кэшировать на уровне target и поддерживать короткоживущие CDP-сессии.
- Ограничения relay расширения:
  - Митигирование: использовать API подключения на уровне браузера, когда сокеты на уровне страницы недоступны, и сохранять текущий путь Playwright как fallback.
